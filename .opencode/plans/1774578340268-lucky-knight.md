# Plan: PM2.5 AQI Fan Automation for Home Assistant

## Context

- AQI sensor: `sensor.indoor_aqi` (template sensor, integer 0–500)
- Fan: `fan.air_purifier`
- Turn on: ~85 AQI (use 85 as threshold)
- Turn off: ~70 AQI (hysteresis band of 15)
- Respect manual off: yes — if user turns fan off while AQI is still high, don't re-engage until AQI recovers and rises again
- File to create: `/tmp/aqi_fan_automation.yaml`

## Design Decisions

### Threshold values
- **On threshold: 85** (crossing into "Unhealthy for Sensitive Groups" territory)
- **Off threshold: 70** (provides a 15-point hysteresis band to prevent rapid cycling)

### Architecture: two separate automations + startup handler

The most idiomatic HA approach for this kind of threshold control is **two `numeric_state` trigger automations** (one for on, one for off) plus a **third startup automation** to reconcile state on HA restart.

Why not one automation with `choose`:
- `numeric_state` triggers inherently track threshold-crossing direction; splitting on/off into two automations is the direct expression of that
- Each automation's trigger has independent state tracking

Why not a template binary sensor + single automation:
- Self-referencing templates have initialization ambiguity on startup
- Two automations is the documented, community-standard pattern

### Manual override: context-based, not a helper entity

The `trigger.to_state.context.user_id` approach is cleaner than an `input_boolean` helper:
- No extra entity to manage
- No persistent state to go stale
- Works by detecting "a human turned the fan off while AQI was still high" → skip re-engagement until AQI drops below the off threshold and rises again

Implementation: use a `variable` set in an automation to track override, OR use the cleaner approach of simply **not having the turn-on automation re-fire** after a manual-off, since `numeric_state` only fires on threshold *crossing*, not continuously. Because `numeric_state` fires once on crossing, a manual off while AQI is still high will NOT retrigger the on-automation (AQI never re-crossed the 85 threshold). This is the key insight — the override problem is already solved by `numeric_state` semantics!

The only scenario needing extra handling: startup. On restart, if AQI is above 85 but the fan was manually turned off before the restart, the startup automation would incorrectly turn the fan on. To handle this:
- On startup, only turn on the fan if it is *currently off* AND AQI > 85, and do NOT override if the fan was last changed by a human (check `last_changed` recency is insufficient; instead, just don't touch it if it's already off — a manual off before restart is indistinguishable from desired off state, so **on startup, only turn on if AQI > 85 AND fan is currently off**; do not turn off automatically at startup to avoid fighting with the user).

Actually the cleanest startup policy: **on startup, if AQI > 85 and fan is off, turn on; if AQI < 70 and fan is on (and automation was last actor), turn off. If in the hysteresis band (70–85), leave as-is.**

But we can't know "automation was last actor" at startup without a helper. Instead: **only turn fan ON at startup** (never force it off at startup). This is the least-surprising behavior:
- AQI high + fan off → turn on (correct)
- AQI high + fan on → leave on (correct)
- AQI low + fan on → leave on (won't cycle off until next numeric_state crossing event)
- AQI low + fan off → leave off (correct)

The startup automation will fully reconcile: turn on if AQI > 85 + fan off, AND turn off if AQI < 70 + fan on. If AQI is in the hysteresis band (70–85), leave as-is.

### `for:` duration

Not used for the primary hysteresis (threshold values handle that), but a short `for: minutes: 2` on the turn-on trigger prevents false triggers from brief spikes. The `for:` timer not surviving restarts is not a problem here — the startup automation handles re-evaluation at restart, and `for: 2 minutes` is short enough that losing a pending timer is acceptable.

### Unavailability guard

`sensor.indoor_aqi` is a template sensor with an `availability:` template, so it will properly be `unavailable` when the source sensor is unavailable. `numeric_state` triggers do not fire when the source is unavailable. The startup automation must guard with `has_value()`.

## Final Automation Structure

### Automation 1: Turn on fan when AQI is high
```yaml
- alias: "Air Purifier: Turn on when AQI is high"
  description: >
    Turns on the air purifier when indoor AQI rises above 85 and stays there
    for 2 minutes (filters transient spikes). numeric_state only fires on
    threshold crossing, so a manual off will not re-trigger until AQI first
    drops below 85 and rises again.
  triggers:
    - trigger: numeric_state
      entity_id: sensor.indoor_aqi
      above: 85
      for:
        minutes: 2
  conditions:
    - condition: state
      entity_id: fan.air_purifier
      state: "off"
  actions:
    - action: fan.turn_on
      target:
        entity_id: fan.air_purifier
  mode: single
```

### Automation 2: Turn off fan when AQI recovers
```yaml
- alias: "Air Purifier: Turn off when AQI recovers"
  description: >
    Turns off the air purifier when AQI drops below 70 (15-point hysteresis
    below the 85 on-threshold). Only turns off if the fan is currently on.
  triggers:
    - trigger: numeric_state
      entity_id: sensor.indoor_aqi
      below: 70
  conditions:
    - condition: state
      entity_id: fan.air_purifier
      state: "on"
  actions:
    - action: fan.turn_off
      target:
        entity_id: fan.air_purifier
  mode: single
```

### Automation 3: Reconcile on HA startup
```yaml
- alias: "Air Purifier: Reconcile AQI state on startup"
  description: >
    On HA restart, fully reconciles fan state with current AQI.
    Turns on if AQI > 85 and fan is off.
    Turns off if AQI < 70 and fan is on.
    Leaves fan unchanged if AQI is in the hysteresis band (70–85).
  triggers:
    - trigger: homeassistant
      event: start
  conditions:
    - condition: template
      value_template: "{{ has_value('sensor.indoor_aqi') }}"
  actions:
    - choose:
        - conditions:
            - condition: numeric_state
              entity_id: sensor.indoor_aqi
              above: 85
            - condition: state
              entity_id: fan.air_purifier
              state: "off"
          sequence:
            - action: fan.turn_on
              target:
                entity_id: fan.air_purifier
        - conditions:
            - condition: numeric_state
              entity_id: sensor.indoor_aqi
              below: 70
            - condition: state
              entity_id: fan.air_purifier
              state: "on"
          sequence:
            - action: fan.turn_off
              target:
                entity_id: fan.air_purifier
  mode: single
```

## File to Create

`/tmp/aqi_fan_automation.yaml` — standalone automation YAML block, suitable for pasting into `configuration.yaml` under `automation:` or into a separate automations file.

## Edge Cases Handled

| Scenario | Handling |
|---|---|
| AQI spikes briefly above 85 | `for: minutes: 2` on turn-on trigger absorbs transient spikes |
| AQI bounces around 85 | `numeric_state` only fires on true threshold crossing, not on every update |
| Fan manually turned off while AQI still high | `numeric_state` won't re-fire until AQI drops below 85 and rises again |
| HA restarts while AQI is high | Startup automation turns fan on if it's off |
| HA restarts while AQI is low, fan is on | Startup automation turns fan off (AQI < 70 + fan on) |
| `sensor.indoor_aqi` is unavailable | `numeric_state` won't fire; startup automation guards with `has_value()` |
| AQI in hysteresis band (70–85) | Neither automation fires; fan stays in whatever state it's in |
| Fan already on when AQI crosses 85 | Turn-on condition `state: off` prevents duplicate action |
| Fan already off when AQI crosses 70 | Turn-off condition `state: on` prevents duplicate action |
